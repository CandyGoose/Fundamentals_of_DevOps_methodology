## 1 часть

В [Dockerfile.bad](./Dockerfile.bad) написаны плохие практики, а в [Dockerfile.good](./Dockerfile.good) они исправлены.

### Лишние пакеты

```Dockerfile
RUN apt-get update && apt-get install -y \
    openssh-server \
    telnet \
    vim \
    curl
```

Если так сделать, то размер образа увеличится, а также увеличится поверхность атак. Соответственно, лучше убрать всё, что мы не собираемся использовать, чтобы уменьшить размер образа и избежать уязвимостей, которые могут принести с собой пакеты.

### Секрет в ENV

```Dockerfile
ENV DB_PASSWORD=super_secret_password
```

Пароль попадет в слои образа и может утечь. А если мы еще и запушили его, например на GitHub, то будет проще поменять пароль, чем исправить ситуацию и изъять его из истории. Исправляем, передавая секреты при запуске контейнера, то есть уменьшаем риск утечки конфиденциальных данных.

### Лишние порты

```Dockerfile
EXPOSE 22 80 443 8080
```

Все порты, кроме 8080, нашему приложению не нужны. Более того, мы открыли 22 порт (SSH), то есть открываем лишние точки входа. Всё это увеличивает поверхность атаки. Исправляем, оставляя только нужный порт и уменьшая точки входа:

```Dockerfile
EXPOSE 8080
```

### Публикация порта наружу без необходимости (работа с контейнерами)

```bash
docker run -p 8080:8080 myapp
```

Если так запустить контейнер, сервис будет доступен не только локально, но и снаружи хоста. Для локальной разработки это обычно не нужно и повышает риск. Исправляем, ограничивая публикацию порта только localhost:

```bash
docker run -p 127.0.0.1:8080:8080 myapp
```

То есть сервис остается доступным только локально, риск несанкционированного внешнего доступа снижается.

### Запуск контейнера без лимитов ресурсов (работа с контейнерами)

```bash
docker run myapp
```

Если запускать контейнер без ограничений, он может занять слишком много памяти или CPU и повлиять на работу других сервисов на хосте. Исправим, задав лимиты ресурсов:

```bash
docker run --memory=256m --cpus=1 myapp
```

Таким образом, контейнер потребляет предсказуемый объем ресурсов и меньше мешает работе других приложений.

## 2 часть

В [docker-compose.bad.yml](./docker-compose.bad.yml) написаны плохие практики, а в [docker-compose.good.yml](./docker-compose.good.yml) они исправлены.

### Использование latest

```yaml
image: nginx:latest
image: redis:latest
```

Этот тег не фиксирует версию. Через время можно получить другой образ и непредсказуемое поведение.  
Исправляем, указывая конкретные версии, делая сборку и запуск воспроизводимыми:

```yaml
image: nginx:1.27-alpine
image: redis:7.4-alpine
```

### Запуск сервисов с избыточными привилегиями

```yaml
privileged: true
```

Это почти снимает защитные ограничения Docker и дает контейнеру слишком много прав. Если приложение внутри контейнера уязвимо, атакующему будет проще выйти за пределы контейнера и повлиять на хост. Исправляем, убирая privileged у сервисов, чтобы контейнер остался более изолированным, а риск компрометации хоста уменьшился.

### Нет volume для данных БД

```yaml
services:
  db:
    image: redis:latest
```

Если не подключить volume для базы, данные внутри контейнера могут пропасть после удаления или пересоздания контейнера. Исправляем, подключая именованный volume:

```yaml
services:
  db:
    image: redis:7.4-alpine
    volumes:
      - redis_data:/data
```

Теперь наши данные Redis сохраняются между перезапусками и пересозданием контейнера.

### Достигаем изоляции

app подключаем только к app_net, db подключаем к db_net, обе сети помечаем internal: true, то есть общей сети у них нет.

```yaml
services:
  app:
  ...
  networks:
    # Отдельная сеть для app
    - app_net

  db:
  ...
  networks:
    # Отдельная сеть для db
    - db_net

networks:
  app_net:
    driver: bridge
    # Внутренняя сеть без внешнего доступа
    internal: true
  db_net:
    driver: bridge
    # Внутренняя сеть без внешнего доступа
    internal: true
```

То есть оба контейнера запускаются в одном compose, но оказываются в разных сетевых сегментах. Docker DNS и маршрутизация работают только внутри конкретной сети, поэтому app не сможет обратиться к db по имени db, как и db не сможет обратиться к app по имени app.

## Выводы

В ходе лабораторной работы были рассмотрены типичные ошибки при работе с Dockerfile и Docker Compose, а также показаны их исправления.

По Dockerfile:

- Отказ от лишних пакетов уменьшает размер образа и поверхность атаки.
- Вынесение секретов из ENV снижает риск утечки конфиденциальных данных.
- Открытие только нужных портов делает контейнер безопаснее.

По работе с контейнерами:

- Ограничение публикации портов уменьшает риск внешнего несанкционированного доступа.
- Лимиты CPU и памяти повышают стабильность хоста.

По Docker Compose:

- Фиксированные версии образов дают воспроизводимые сборки.
- Отказ от privileged: true снижает риск компрометации хоста.
- Подключение volume для БД сохраняет данные между перезапусками.

Также настроена сетевая изоляция сервисов в хорошем compose: контейнеры запускаются вместе, но не имеют прямого сетевого доступа друг к другу, что повышает безопасность и способствует минимальному необходимому взаимодействию между сервисами.
